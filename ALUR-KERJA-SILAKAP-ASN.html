1. RINGKASAN EKSEKUTIF

SILAKAP-ASN (Sistem Informasi Layanan Kepegawaian ASN) adalah sistem manajemen layanan kepegawaian berbasis web yang dikembangkan untuk mendukung operasional BKPSDM Kabupaten Tolitoli. Sistem ini dirancang sebagai platform workflow-driven, data-driven, dan audit-ready — bukan sekadar kumpulan menu atau formulir input.

Aspek Keterangan
Tujuan Utama Mengotomatiskan dan melacak proses layanan kepegawaian ASN secara end-to-end
Pengguna Utama ASN, Operator BKPSDM, Verifikator, PPK, Admin BKPSDM, Super Admin
Layanan Utama Pensiun, Mutasi, KGB, Jabatan, Peremajaan Data
Teknologi Backend NestJS 10, Prisma ORM, BullMQ, Redis, OpenTelemetry
Teknologi Frontend React 18, Vite, TypeScript, TanStack Query, React Hook Form
Database MySQL (via Prisma ORM)
Autentikasi JWT (JSON Web Token) + Role-Based Access Control
Integrasi SIASN (Sistem Informasi ASN Nasional) via sinkronisasi data
Prinsip Utama Pengembangan:
Alur bisnis lebih penting daripada kosmetik UI. Source of truth harus jelas. Satu perubahan workflow dianggap perubahan lintas layer.

2. ARSITEKTUR SISTEM

Sistem SILAKAP-ASN dibangun dalam tiga lapisan arsitektur utama yang saling bergantung secara berurutan:

2.1 Control Plane — Lapisan Kendali

Lapisan ini adalah fondasi sistem. Tanpa lapisan ini, transaksi layanan akan liar dan tidak konsisten.

Komponen Fungsi
Master Referensi Menyediakan data referensi stabil: jenis layanan, jabatan, organisasi, dokumen, status kepegawaian
Pengaturan Role Mendefinisikan siapa boleh melihat apa dan mengerjakan apa
Workflow Engine Mengatur transisi status layanan dan aturan persetujuan
Permission Matrix Matriks hak akses per role ke setiap menu dan aksi sistem

2.2 Transaction Plane — Lapisan Transaksi

Lapisan ini adalah inti aplikasi yang dipakai pengguna untuk bekerja sehari-hari.

Komponen Fungsi
Data ASN Single source of truth identitas pegawai, jabatan, unit kerja, status, riwayat
Layanan ASN Proses pengajuan: create, draft, submit, workflow, detail
Verifikasi & Persetujuan Antrian kerja, tindakan workflow, disposisi
Dokumen & Arsip Upload dokumen, validasi kelengkapan, arsip digital

2.3 Visibility & Governance Plane — Lapisan Keterlihatan

Lapisan ini dibangun setelah proses inti stabil.

Komponen Fungsi
Dashboard & Laporan Ringkasan layanan, SLA warning, laporan pegawai, export Excel/PDF
Keamanan & Audit Audit log, aktivitas pengguna, ketertelusuran perubahan
Integrasi Eksternal Sinkronisasi SIASN, import data referensi, monitoring job
Analitik SDM Dashboard SDM, formasi, ABK, talent pool

3. DEPENDENCY MAP & URUTAN IMPLEMENTASI

Berikut adalah urutan ketergantungan antar modul. Setiap modul hanya dapat berfungsi optimal jika modul di atasnya sudah stabil:

MASTER REFERENSI
    ↓ (referensi stabil)
PENGATURAN SISTEM (Role + Workflow)
    ↓ (aturan otorisasi jelas)
DATA ASN
    ↓ (identitas pegawai tersedia)
LAYANAN ASN (Pensiun, Mutasi, KGB, Jabatan, Peremajaan)
    ↓ (transaksi berjalan)
VERIFIKASI & PERSETUJUAN
    ↓ (workflow hidup end-to-end)
DOKUMEN & ARSIP
    ↓ (layanan valid administratif)
DASHBOARD & LAPORAN
    ↓ (visibilitas operasional)
KEAMANAN & AUDIT
    ↓ (sistem siap diaudit)
INTEGRASI EKSTERNAL (SIASN)
    ↓ (sinkronisasi data)
ANALITIK SDM

Penjelasan Setiap Dependency

Modul Jika Tidak Ada / Tidak Stabil

Dokumen & Arsip Tidak ada syarat kelengkapan untuk submit dan approve — layanan tidak valid administratif

5. ALUR KERJA LAYANAN ASN

5.1 Alur Umum Layanan (Berlaku untuk Semua Layanan)

Setiap layanan ASN di SILAKAP mengikuti alur baku yang sama. Perbedaan hanya pada data domain spesifik masing-masing layanan.

[ASN/OPERATOR] Buat Usulan (Create)     ↓ [SISTEM] Simpan sebagai DRAFT     ↓ [ASN/OPERATOR] Lengkapi Data & Dokumen     ↓ [ASN/OPERATOR] Submit Usulan     ↓ (validasi dokumen & dependency) [SISTEM] Status: SUBMITTED     ↓ [VERIFIKATOR] Terima di Antrian Verifikasi     ↓ [VERIFIKATOR] Verifikasi Dokumen & Data     ↓ ↓ (dua kemungkinan) [LULUS VERIFIKASI] [DIKEMBALIKAN]     ↓ ↓ [PPK] Persetujuan [OPERATOR] Perbaiki     ↓ ↓ ↓ [DISETUJUI] [DITOLAK] [Re-Submit]     ↓ [SISTEM] Buat Dokumen Hasil     ↓ [SELESAI] Arsip & Audit Trail Lengkap

Tabel Status Workflow

Status Keterangan Aktor
DRAFT Usulan dibuat, belum disubmit. Dapat diedit. ASN / Operator
SUBMITTED Usulan disubmit, menunggu verifikasi. Tidak dapat diedit. ASN / Operator
VERIFIED Dokumen dan data sudah diverifikasi, menunggu persetujuan PPK. Verifikator
RETURNED Dikembalikan karena ada kekurangan. ASN dapat memperbaiki. Verifikator
APPROVED Disetujui oleh PPK. Proses selesai. PPK
REJECTED Ditolak. Usulan tidak dapat dilanjutkan. PPK / Verifikator
COMPLETED Dokumen hasil diterbitkan dan diarsip. Sistem / Admin

5.2 Layanan Pensiun

Pensiun adalah layanan flagship dan blueprint arsitektur untuk semua layanan lain di sistem ini.

Jenis Pensiun

  • BUP (Batas Usia Pensiun) — pensiun karena mencapai batas usia
  • APS (Atas Permintaan Sendiri) — pensiun sukarela
  • Pensiun Dini — pensiun sebelum BUP karena kondisi tertentu
  • Pensiun Janda/Duda — pensiun ahli waris

Alur Detail Layanan Pensiun

No. Tahap Aktor Tindakan
1 Buat Usulan Operator BKPSDM Input data pensiun: NIP, jenis pensiun, TMT pensiun, alasan
2 Simpan Draft Sistem Resolve pegawaiId, jenisLayananId, simpan ke database dengan status DRAFT
3 Upload Dokumen Operator / ASN Upload SK terakhir, SK Pangkat, ijazah, surat keterangan, dll.
4 Submit Operator Validasi kelengkapan dokumen wajib, ubah status ke SUBMITTED
5 Verifikasi Verifikator Cek kesesuaian data dan kelengkapan dokumen. Lulus → VERIFIED. Kurang → RETURNED.
6 Persetujuan PPK PPK Review usulan yang telah diverifikasi. Setuju → APPROVED. Tolak → REJECTED.
7 Proses BKN/BPD Admin BKPSDM Pengiriman berkas ke BKN/BPD untuk penetapan SK Pensiun
8 Selesai Sistem Update status COMPLETED, simpan audit trail, arsip dokumen

Dokumen Wajib Pensiun

  • SK Pangkat/Golongan Terakhir
  • SK Jabatan Terakhir
  • Surat Keterangan Bebas Temuan (dari Inspektorat)
  • Fotocopy Ijazah Terakhir
  • Fotocopy KTP & KK
  • Pas Foto 4x6 (terbaru)
  • Daftar Riwayat Pekerjaan (DRH)

5.3 Layanan Mutasi

Proses perpindahan jabatan atau unit kerja pegawai, baik dalam satu instansi maupun antar instansi.

Jenis Mutasi

  • Mutasi Internal — perpindahan antar unit dalam satu instansi
  • Mutasi Eksternal (Masuk) — penerimaan pegawai dari instansi lain
  • Mutasi Eksternal (Keluar) — pelepasan pegawai ke instansi lain

Alur Layanan Mutasi

No. Tahap Aktor Tindakan
1 Inisiasi Operator / PPK Input data mutasi: NIP, jabatan asal, jabatan tujuan, UNOR tujuan, TMT
2 Draft Sistem Simpan usulan mutasi dengan status DRAFT
3 Dokumen Operator Upload surat permohonan, persetujuan atasan, nota dinas
4 Submit Operator Validasi kelengkapan, submit ke Verifikator
5 Verifikasi Verifikator Cek kesesuaian formasi, kebutuhan unit, dan persyaratan
6 Persetujuan PPK Setujui atau tolak usulan mutasi
7 SK Mutasi Admin BKPSDM Penerbitan SK Mutasi, update data pegawai

5.4 Layanan Kenaikan Gaji Berkala (KGB)

Proses kenaikan gaji berkala yang terjadi setiap 2 tahun sekali bagi pegawai yang memenuhi syarat.

Alur Layanan KGB

No. Tahap Aktor Tindakan
1 Identifikasi Sistem / Operator Identifikasi pegawai yang memenuhi syarat KGB (masa kerja 2 tahun dari KGB terakhir)
2 Buat Usulan Operator Input: NIP, gaji pokok lama, golongan, masa kerja, TMT KGB baru
3 Dokumen Operator Upload SK KGB terakhir, SKP 2 tahun terakhir, surat pengantar
4 Submit & Verifikasi Operator / Verifikator Submit, verifikasi data dan dokumen
5 Persetujuan PPK PPK Setujui usulan KGB
6 SK KGB Admin Penerbitan SK KGB, update gaji pokok pegawai di data ASN

5.5 Layanan Jabatan

Proses pengangkatan, pelantikan, dan perpindahan jabatan struktural maupun fungsional.

Jenis Layanan Jabatan

  • Pengangkatan Jabatan Struktural
  • Pengangkatan Jabatan Fungsional
  • Kenaikan Jenjang Jabatan Fungsional
  • Pemberhentian dari Jabatan
  • Pelaksana Tugas (Plt)

Alur Layanan Jabatan

No. Tahap Aktor Tindakan
1 Inisiasi PPK / Operator Input data pengangkatan: NIP, jabatan baru, TMT, dasar hukum
2 Validasi Syarat Sistem Cek persyaratan jabatan (golongan minimal, pendidikan, masa kerja)
3 Dokumen Operator Upload berita acara serah terima, keputusan Baperjakat, surat pertimbangan
4 Verifikasi Verifikator Cek kesesuaian formasi dan persyaratan jabatan
5 Persetujuan PPK Setujui pengangkatan
6 SK Jabatan Admin Penerbitan SK Jabatan, update data jabatan pegawai

5.6 Layanan Peremajaan Data

Proses pembaruan data master pegawai yang mengalami perubahan (nama, gelar, pendidikan, dll.).

Alur Layanan Peremajaan Data

No. Tahap Aktor Tindakan
1 Pengajuan ASN / Operator Input field yang ingin diperbarui beserta nilai baru
2 Bukti Perubahan ASN / Operator Upload dokumen pendukung (putusan pengadilan, ijazah baru, dll.)
3 Verifikasi Verifikator Validasi kebenaran perubahan data dan kelengkapan bukti
4 Persetujuan PPK / Admin Setujui pembaruan data
5 Update Data Sistem Update data master pegawai dan sinkronisasi ke SIASN

6. ALUR VERIFIKASI & PERSETUJUAN

6.1 Prinsip Pembagian Kerja

Semua usulan layanan ASN masuk melalui Operator OPD — staf yang ditunjuk di setiap Organisasi Perangkat Daerah (Dinas, Badan, Kantor, Kecamatan). BKPSDM berperan sebagai pengolah dan pemutus, bukan pengumpul data. Di ujung akhir proses, PPPK Paruh Waktu menangani dokumentasi dan pengarsipan hasil yang telah disetujui.

Prinsip Alur: OPD mengajukan → BKPSDM memverifikasi & memutuskan → PPPK mendokumentasikan. Tidak ada usulan yang diinput langsung oleh staf BKPSDM kecuali ada kondisi khusus yang diizinkan Kepala Bidang.

6.7 Audit Trail Wajib per Transisi

Setiap perpindahan status wajib menulis 4 rekam jejak:

  1. Perubahan Status — status baru, jabatan aktor, NIP, timestamp
  2. Layanan Log — catatan keputusan dan catatan yang diberikan
  3. Workflow Timeline — histori alur per tahap dengan durasi proses
  4. Audit Log — rekam jejak sistem untuk keperluan audit BPK/Inspektorat
Jika salah satu dari keempat hal di atas tidak tertulis, workflow dianggap belum selesai dan tidak memenuhi Definition of Done. Setiap aksi harus disertai catatan/komentar terutama untuk aksi RETURN dan REJECT.

7.2 Rincian Tugas Harian per Jabatan

Operator OPD  (staf di setiap Dinas/Badan/Kantor/Kecamatan)

  • Menerima berkas fisik dari ASN pegawai OPD-nya
  • Menscan/memfoto dokumen persyaratan
  • Menginput data usulan ke sistem SILAKAP melalui akun OPD
  • Mengupload dokumen hasil scan ke sistem
  • Mengsubmit usulan ke BKPSDM
  • Memantau status usulan yang telah disubmit
  • Menerima notifikasi dan memperbaiki berkas yang dikembalikan BKPSDM
  • Mengambil dan mendistribusikan dokumen hasil (SK, dll.) ke ASN yang bersangkutan

Pelaksana Teknis BKPSDM  (internal BKPSDM)

  • Memantau volume antrian usulan yang masuk dari seluruh OPD
  • Menjadi kontak pertama OPD jika ada kendala teknis dalam penggunaan sistem
  • Membantu OPD yang mengalami kesulitan teknis (bukan mengambil alih input)
  • Memonitor SLA dan melaporkan bottleneck kepada Kepala Bidang
  • Mengelola akun dan akses Operator OPD (pembuatan akun baru, reset password)
  • Mengelola konfigurasi teknis sistem bersama dengan tim IT

Analis SDMA Ahli Pertama  (verifikasi kelengkapan — gate pertama BKPSDM)

  • Membuka antrian verifikasi kelengkapan setiap pagi hari kerja
  • Memeriksa daftar dokumen wajib berdasarkan jenis layanan yang diajukan
  • Memeriksa keabsahan dokumen: tanda tangan, cap dinas, tanggal, legibilitas
  • Mengembalikan berkas yang kurang dengan catatan rinci dan spesifik tentang apa yang kurang
  • Meneruskan berkas yang lengkap ke Analis Ahli Muda
  • Melakukan verifikasi mandiri untuk layanan Peremajaan Data (verifikasi sekaligus substansi)

Analis SDMA Ahli Muda  (verifikasi substansi — gate teknis utama)

  • Menerima berkas yang sudah dinyatakan lengkap oleh Analis Ahli Pertama
  • Memverifikasi kesesuaian data dengan peraturan perundang-undangan yang berlaku
  • Memeriksa syarat teknis: masa kerja, nilai SKP, golongan, syarat usia, syarat jabatan
  • Meneruskan berkas yang valid ke tahap berikutnya
  • Mengembalikan berkas yang tidak memenuhi syarat dengan dasar hukum yang jelas
  • Menjadi verifikator final untuk layanan KGB dan Jabatan (teruskan ke Kepala Bidang)
  • Melaporkan pola kesalahan berulang dari OPD tertentu kepada Analis Madya

Analis SDMA Ahli Madya  (telaahan teknis — hanya Pensiun & Mutasi)

  • Menerima berkas Pensiun dan Mutasi yang sudah valid dari Analis Ahli Muda
  • Melakukan telaahan teknis mendalam terhadap substansi dan dampak usulan
  • Menyusun nota pertimbangan teknis tertulis untuk setiap usulan
  • Menganalisis preseden, risiko, dan konsistensi dengan kebijakan yang berlaku
  • Memberikan rekomendasi tertulis (setuju/tidak setuju beserta alasan) kepada Kepala Bidang
  • Mengkoordinasikan isu yang memerlukan konfirmasi lintas bidang
  • Membimbing Analis Ahli Muda dan Pertama dalam penyelesaian kasus teknis yang kompleks

Kepala Bidang  (persetujuan teknis bidang)

  • Mereview nota pertimbangan teknis dari Analis Madya untuk Pensiun dan Mutasi
  • Merekomendasikan ke Kepala Badan untuk layanan Pensiun dan Mutasi
  • Memberikan persetujuan final untuk KGB, Jabatan, dan Peremajaan Data
  • Menandatangani dokumen hasil yang sesuai batas kewenangannya
  • Monitoring SLA tim dan bottleneck proses verifikasi
  • Melaporkan rekapitulasi layanan kepada Kepala Badan secara berkala

Kepala Badan  (penetapan final — hanya Pensiun & Mutasi)

  • Menerima usulan Pensiun dan Mutasi yang sudah direkomendasikan Kepala Bidang
  • Memberikan penetapan/persetujuan final atau penolakan beserta catatan
  • Menandatangani SK Pensiun dan SK Mutasi
  • Memonitor dashboard eksekutif: SLA keseluruhan, volume layanan, OPD paling aktif
  • Tidak terlibat dalam proses teknis harian — fokus pada keputusan strategis

PPPK Paruh Waktu  (dokumentasi & pengarsipan — bekerja di ujung proses)

  • Memantau antrian usulan yang berstatus APPROVED setiap hari
  • Mencetak SK dan dokumen hasil yang telah ditandatangani pejabat berwenang
  • Menscan dan mengupload dokumen hasil ke sistem DMS (arsip digital)
  • Menyimpan arsip fisik ke dalam ordner/lemari arsip sesuai klasifikasi
  • Mencatat nomor SK dan data layanan ke buku register/ekspedisi
  • Menyiapkan dokumen untuk didistribusikan ke OPD pemohon
  • Mengubah status usulan di sistem menjadi COMPLETED setelah semua dokumentasi selesai
  • Bekerja pada jam paruh waktu yang ditetapkan (misal: 07.30–12.00)

8. INTEGRASI EKSTERNAL — SIASN

8.1 Gambaran Integrasi

SILAKAP-ASN terhubung dengan sistem nasional SIASN (Sistem Informasi ASN) milik Badan Kepegawaian Negara (BKN) untuk memastikan data kepegawaian tetap sinkron dan valid secara nasional.

8.2 Alur Import Data Referensi (Excel)

Data referensi (jabatan dan UNOR) diimport dari file Excel yang diekspor dari SIASN.

No. Tahap Keterangan
1 Download dari SIASN Admin mendownload file Excel data referensi dari portal SIASN
2 Upload ke SILAKAP Upload melalui menu Integrasi Eksternal → Import Data
3 Parsing & Validasi Sistem membaca file Excel, validasi format, filter baris tidak valid
4 Upsert Database Data yang valid di-upsert: create jika baru, update jika sudah ada (berdasarkan ID SIASN)
5 Laporan Hasil Sistem menampilkan: total baris, baris valid, created, updated, skipped

11. CHECKLIST & DEFINITION OF DONE

11.1 Checklist Verifikasi Setelah Setiap Perubahan

No. Checklist Item Status
1 cd api && npm run build — Backend build bersih tanpa error ☐
2 cd web && npm run build — Frontend build bersih tanpa error TypeScript ☐
3 Create usulan berfungsi (data tersimpan ke database) ☐
4 Submit usulan berfungsi (validasi dokumen + ubah status) ☐
5 Halaman detail usulan menampilkan data lengkap ☐
6 Tindakan workflow utama berhasil dieksekusi ☐
7 Timeline dan log workflow terisi lengkap ☐
8 Halaman laporan menampilkan data nyata (bukan dummy) ☐
9 Export laporan berfungsi tanpa merusak halaman utama ☐

11.2 Definition of Done — Suatu Pekerjaan Dinyatakan Selesai Jika

No. Kriteria
1 Endpoint aktif dan terdokumentasi jelas
2 Payload dan response frontend-backend sudah sinkron
3 Role akses sudah sesuai (tidak ada akses yang bocor atau terblokir salah)
4 UI tidak crash di browser (tidak ada error konsol yang kritis)
5 Build api dan web tetap hijau (tidak ada error TypeScript)
6 Audit trail lengkap: status, layanan log, workflow timeline, audit log
7 Laporan tidak memakai data dummy untuk runtime production
8 Tidak ada penggunaan route legacy yang baru dibuat

11.3 Panduan Troubleshooting Cepat

Gejala Yang Harus Dicek
Tombol workflow ada tapi gagal dieksekusi 1) Endpoint aktif benar  2) Payload ada usulId & actionCode  3) JWT valid  4) Role cocok  5) Status usul cocok dengan transisi yang diijinkan
Create usulan gagal 1) Service code valid  2) pegawaiId/NIP bisa di-resolve  3) Payload domain lengkap  4) Referensi bisa dinormalisasi
Timeline/log kosong 1) Backend menulis layananLog  2) Workflow menulis log  3) Workflow menulis timeline  4) Workflow menulis audit
Laporan kosong 1) Query tabel sumber benar  2) Klasifikasi tidak terlalu sempit  3) Filter UNOR aktif  4) Backend sudah restart
Export gagal 1) Endpoint export aktif  2) Fallback frontend aman  3) Popup browser tidak diblokir  4) Format file cocok dengan ekstensi
Upload file error "Duplicate entry" 1) Tabel punya data lama dengan kode konflik  2) Upload ulang setelah fix — sistem akan upsert otomatis berdasarkan ID SIASN
